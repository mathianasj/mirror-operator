# Keycloak TLS Configuration

This document explains how to configure TLS certificates for the managed Keycloak instance in the mirror-operator.

## Overview

When using managed Keycloak for RHTAS OIDC authentication, TLS certificates are required. The mirror-operator supports two approaches:

1. **cert-manager Issuer (Recommended)**: Specify a ClusterIssuer or Issuer for automatic certificate management
2. **Custom TLS Secret**: Bring your own certificates

## Option 1: cert-manager Issuer (Recommended)

### Overview

The operator uses cert-manager to automatically create and manage TLS certificates. You must specify which ClusterIssuer or Issuer to use.

### Prerequisites

- cert-manager operator installed
- A ClusterIssuer or Issuer configured (e.g., Let's Encrypt, corporate CA)

### Configuration

```yaml
rhtas:
  oidc:
    managed:
      enabled: true
      certIssuer:
        name: letsencrypt-prod  # Your ClusterIssuer name
        kind: ClusterIssuer     # or "Issuer" for namespace-scoped
```

### Examples

**Use Let's Encrypt Production**:
```yaml
certIssuer:
  name: letsencrypt-prod
  kind: ClusterIssuer
```

**Use Corporate CA**:
```yaml
certIssuer:
  name: corporate-ca-issuer
  kind: ClusterIssuer
```

**Use Namespace-scoped Issuer**:
```yaml
certIssuer:
  name: my-ca-issuer
  kind: Issuer  # Must be in mirror-operator-system namespace
```

### What Gets Created

When you specify a `certIssuer`, the operator creates:

**Certificate**:
```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: keycloak-certificate
  namespace: mirror-operator-system
spec:
  secretName: keycloak-tls
  issuerRef:
    name: letsencrypt-prod  # Your specified issuer
    kind: ClusterIssuer
  commonName: keycloak.apps.cluster-example.com
  dnsNames:
    - keycloak.apps.cluster-example.com
  duration: 8760h      # 1 year
  renewBefore: 720h    # 30 days
```

**TLS Secret** (auto-generated by cert-manager):
- Name: `keycloak-tls`
- Type: `kubernetes.io/tls`
- Contains: `tls.crt` and `tls.key`

### Verification

```bash
# Check the certificate
kubectl get certificate keycloak-certificate -n mirror-operator-system -o yaml | grep issuerRef -A 2

# Verify the certificate is ready
kubectl get certificate keycloak-certificate -n mirror-operator-system

# Check the secret
kubectl get secret keycloak-tls -n mirror-operator-system
```

### Advantages

- ✅ Uses your cluster's existing certificate infrastructure
- ✅ Automatic certificate management and renewal
- ✅ Trusted certificates if using Let's Encrypt or corporate CA
- ✅ Consistent with Kubernetes best practices

## Option 2: Custom TLS Secret

### Use Cases

- You have existing CA-signed certificates
- Corporate PKI requirements
- Production environments requiring trusted certificates
- Wildcard certificates

### Prerequisites

- Valid TLS certificate and private key
- Certificate must be valid for the Keycloak hostname: `keycloak.<cluster-domain>`

### Step 1: Create the TLS Secret

```bash
# Create secret from certificate files
kubectl create secret tls my-keycloak-tls \
  --cert=path/to/tls.crt \
  --key=path/to/tls.key \
  -n mirror-operator-system

# Or from a YAML file
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Secret
metadata:
  name: my-keycloak-tls
  namespace: mirror-operator-system
type: kubernetes.io/tls
data:
  tls.crt: $(base64 < tls.crt)
  tls.key: $(base64 < tls.key)
EOF
```

### Step 2: Reference in DisconnectedPlatform

```yaml
apiVersion: mirror.mirror.mathianasj.github.com/v1
kind: DisconnectedPlatform
metadata:
  name: disconnected-platform
spec:
  mode: connected
  connected:
    operators:
      keycloak: {}
      rhtas: {}
    rhtas:
      oidc:
        managed:
          enabled: true
          realm: "trusted-artifact-signer"
          tlsSecret:
            name: my-keycloak-tls  # Reference your custom secret
        type: "email"
```

### Verification

```bash
# Check Keycloak is using your secret
kubectl get keycloak mirror-operator-keycloak -n mirror-operator-system \
  -o jsonpath='{.spec.http.tlsSecret}'
# Should output: my-keycloak-tls

# Check Keycloak pod is running
kubectl get pod -n mirror-operator-system -l app=keycloak
```

### Advantages

- ✅ Use your own CA-signed certificates
- ✅ No browser warnings
- ✅ Meets corporate security requirements
- ✅ Can use wildcard certificates
- ✅ No cert-manager dependency

### Disadvantages

- ⚠️ Manual certificate management
- ⚠️ Manual renewal before expiration
- ⚠️ Must create secret before deploying

## Certificate Renewal

### Automatic (cert-manager)

Cert-manager automatically renews certificates 30 days before expiration. The operator requires no intervention.

### Custom Secrets

When your custom certificate is approaching expiration:

1. Obtain the renewed certificate
2. Update the secret:
   ```bash
   kubectl create secret tls my-keycloak-tls \
     --cert=new-tls.crt \
     --key=new-tls.key \
     -n mirror-operator-system \
     --dry-run=client -o yaml | kubectl apply -f -
   ```
3. Restart Keycloak to pick up the new certificate:
   ```bash
   kubectl rollout restart statefulset/mirror-operator-keycloak \
     -n mirror-operator-system
   ```

## Troubleshooting

### Keycloak Pod CrashLoopBackOff

**Symptom**: Pod logs show "Key material not provided to setup HTTPS"

**Cause**: TLS secret doesn't exist or is malformed

**Solution**:
```bash
# Check if secret exists
kubectl get secret keycloak-tls -n mirror-operator-system

# For cert-manager approach, check certificate status
kubectl get certificate keycloak-certificate -n mirror-operator-system

# Check for cert-manager errors
kubectl describe certificate keycloak-certificate -n mirror-operator-system
```

### Certificate Not Ready

**Symptom**: Certificate shows `READY: False`

**Cause**: cert-manager issues creating the certificate

**Solution**:
```bash
# Check certificate events
kubectl describe certificate keycloak-certificate -n mirror-operator-system

# Check cert-manager logs
kubectl logs -n cert-manager-operator -l app=cert-manager

# Common issues:
# 1. cert-manager operator not installed
# 2. Issuer not ready
# 3. RBAC permissions missing
```

### Hostname Mismatch

**Symptom**: Certificate valid but browsers still show warnings

**Cause**: Certificate hostname doesn't match Keycloak route

**Solution**:
```bash
# Check the actual Keycloak hostname
kubectl get keycloak mirror-operator-keycloak -n mirror-operator-system \
  -o jsonpath='{.spec.hostname.hostname}'

# Check certificate DNS names
kubectl get certificate keycloak-certificate -n mirror-operator-system \
  -o jsonpath='{.spec.dnsNames}'

# These should match!
```

### cert-manager Not Installed

**Symptom**: Certificate resources show `no matches for kind "Certificate"`

**Cause**: cert-manager CRDs not installed

**Solution**:
Either:
1. Install cert-manager operator, OR
2. Use custom TLS secret approach instead

## Production Recommendations

### For Internal/Development Clusters
- Use cert-manager with a self-signed ClusterIssuer
- Simple, fully automated
- Acceptable for internal use

### For Production Clusters
- Use cert-manager with a real CA issuer (Let's Encrypt, corporate CA)
- Or use custom TLS secret with CA-signed certificates if cert-manager isn't available
- Set up monitoring for certificate expiration
- Document renewal procedures

### Enterprise PKI Integration

If you have an enterprise PKI:

1. Request a certificate for `keycloak.<cluster-domain>` from your CA
2. Create the TLS secret as shown above
3. Add the CA certificate to OpenShift's trusted CA bundle:
   ```bash
   kubectl create configmap custom-ca \
     --from-file=ca-bundle.crt=corporate-ca.pem \
     -n openshift-config
   
   kubectl patch proxy/cluster \
     --type=merge \
     --patch='{"spec":{"trustedCA":{"name":"custom-ca"}}}'
   ```
